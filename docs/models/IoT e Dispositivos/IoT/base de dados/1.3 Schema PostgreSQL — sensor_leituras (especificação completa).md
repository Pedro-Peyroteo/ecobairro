**Decisão — guardar leituras inválidas:** Mesmo leituras rejeitadas são guardadas com `valida=false` e `motivo_invalida`. Isto é fundamental para diagnóstico de sensores avariados e para auditoria operacional. O volume adicional é negligenciável — leituras inválidas são uma fracção pequena do total.

**Decisão — `nivel_raw_cm` e `nivel_enchimento` separados:** O sensor envia a distância em centímetros até ao conteúdo. O NestJS converte para percentagem usando `altura_instalacao_cm` da config do dispositivo e aplica o `calibracao_offset`. Guardar ambos permite recalibrar historicamente se a configuração do sensor for corrigida.

**Estratégia de particionamento:**
```
sensor_leituras                           ← tabela pai
  ├── sensor_leituras_2025_01             RANGE('2025-01-01', '2025-02-01')
  ├── sensor_leituras_2025_02             RANGE('2025-02-01', '2025-03-01')
  ├── ...  (pg_partman cria 2 meses à frente)
  └── sensor_leituras_YYYY_MM

Retenção 24 meses (RNF-DADOS-01):
  Job mensal BullMQ 'iot.archive':
    1. Verifica partições com mais de 24 meses
    2. pg_partman detach_partition(partição_antiga)
    3. Exporta para tablespace frio (CSV em Object Storage MinIO)
    4. DROP PARTITION após confirmação de export

Índices por partição:
  (ecoponto_id, timestamp_leitura DESC)  → BTREE  → histórico por ecoponto
  (timestamp_leitura)                    → BRIN   → queries temporais
  (dispositivo_id, timestamp_leitura)    → BTREE  → diagnóstico por sensor
  (nivel_enchimento, timestamp_leitura)  → BTREE PARTIAL WHERE valida=true
     AND nivel_enchimento >= 80          → detecção de overflow
  (valida)                               → PARTIAL BTREE WHERE valida=false
                                         → monitorização de erros
```
