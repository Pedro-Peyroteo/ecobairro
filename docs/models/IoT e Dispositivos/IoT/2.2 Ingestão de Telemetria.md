|#|Método|Rota|Descrição|Auth|Fluxo|
|---|---|---|---|---|---|
|T1|`POST`|`/iot/telemetria`|Ingerir leitura de sensor (alta frequência — 10k msg/min)|API Key header `X-Device-Key`|NestJS → 202 imediato → BullMQ `iot.ingest`|
|T2|`POST`|`/iot/telemetria/batch`|Ingerir lote de leituras (sensores com buffer offline)|API Key header `X-Device-Key`|NestJS → 202 imediato → BullMQ `iot.ingest.batch`|
|T3|`POST`|`/iot/heartbeat`|Heartbeat simples sem leitura — confirma conectividade|API Key header `X-Device-Key`|NestJS → Redis `device:heartbeat:{device_id}` + PG update|

**Corpo de T1 (leitura individual):**

```
{
  device_id: string (required),
  ecoponto_id: uuid (required — rejeita se ausente, RF-04),
  timestamp_leitura: ISO8601 (required),
  nivel_raw_cm: integer (optional),
  nivel_enchimento: integer 0-100 (optional — se sensor calcula internamente),
  estado_reportado_sensor: "CHEIO" | "DISPONIVEL" | "AVARIADO" (optional),
  battery_pct: integer 0-100 (optional),
  signal_rssi: integer (optional, dBm),
  temperatura_c: float (optional),
  versao_firmware: string (optional),
  metadata_extra: object (optional)
}
```

**Corpo de T2 (batch — sensores com buffer):**

```
{
  device_id: string (required),
  leituras: [
    {
      ecoponto_id: uuid,
      timestamp_leitura: ISO8601,
      nivel_raw_cm: integer,
      battery_pct: integer,
      signal_rssi: integer
    },
    ...  (máximo 100 leituras por batch)
  ]
}
```

O endpoint T2 é crítico para sensores em zonas com cobertura intermitente — o sensor guarda leituras localmente e envia em batch quando recupera conectividade. As leituras do batch são processadas em ordem cronológica, e apenas a mais recente actualiza o `ecoponto_estado_atual`.
