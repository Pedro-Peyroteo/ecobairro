**Estratégia de particionamento sensor_leituras:**

```
sensor_leituras                        ← tabela pai (sem dados, só estrutura)
  ├── sensor_leituras_2025_01          RANGE('2025-01-01', '2025-02-01')
  ├── sensor_leituras_2025_02          RANGE('2025-02-01', '2025-03-01')
  ├── ...
  └── sensor_leituras_2027_XX         ← drop automático após 24 meses (RNF-DADOS-01)

Gestão: pg_partman cria partições futuras (2 meses à frente)
        e detacha partições antigas após retenção
Após detach: tablespace frio → DROP confirmado
```

**Justificação do BRIN em vez de BTREE:** Com 10.000 msg/min, cada partição mensal acumula ~432 milhões de linhas. Um índice BTREE em `timestamp_leitura` para esse volume ocupa vários GB de RAM. O índice BRIN (Block Range Index), adequado para dados inseridos sequencialmente por tempo, ocupa ~100KB e responde queries de intervalo temporal eficientemente. O trade-off (não suporta equality lookup) é irrelevante — todas as queries nesta tabela são por intervalo.

**Pipeline NOTIFY para IoT:**

```
E11: POST /iot/telemetria (device_id + nivel + timestamp)
  │
  ├── 202 Accepted ao sensor (resposta imediata)
  │
  └── BullMQ job: 'iot.ingest'
        ├── INSERT sensor_leituras (append na partição actual)
        ├── UPSERT ecoponto_estado_atual
        │     ON CONFLICT (ecoponto_id) DO UPDATE SET
        │       nivel_enchimento = excluded.nivel_enchimento,
        │       estado = <calculado vs limiar da zona>,
        │       atualizado_em = now()
        └── PostgreSQL trigger AFTER UPDATE ON ecoponto_estado_atual
              → NOTIFY ecoponto_estado_updated,
                '{"ecoponto_id":"uuid","zona_id":"uuid","estado":"CHEIO"}'
                  ├── NestJS listener → DEL ecoponto:{id}
                  │                  → DEL ecopontos:mapa:zona:{zona_id}
                  └── BullMQ threshold job (se estado = CHEIO)
                        → Alert dispatcher → SMS/push/email
```
