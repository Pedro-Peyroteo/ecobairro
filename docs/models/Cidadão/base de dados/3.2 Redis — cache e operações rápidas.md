#### Pipeline LISTEN/NOTIFY para invalidação reactiva

```
PostgreSQL trigger (AFTER UPDATE ON users)
    │
    └─→ NOTIFY user_updated, '{"id": "uuid"}'
             │
             └─→ NestJS PG pub/sub listener (nestjs-pg-pubsub)
                      │
                      ├─→ DEL user:session:{id}
                      └─→ DEL user:rbac:{id}
```

```
PostgreSQL trigger (AFTER UPDATE ON cidadao_perfis)
    │
    └─→ NOTIFY cidadao_updated, '{"user_id": "uuid"}'
             │
             └─→ NestJS PG pub/sub listener
                      │
                      ├─→ DEL cidadao:profile:{user_id}
                      ├─→ DEL cidadao:notif_prefs:{user_id}
                      ├─→ DEL cidadao:dashboard:{user_id}
                      └─→ DEL cidadao:favoritos:{user_id}
```

```
PostgreSQL trigger (AFTER UPDATE ON operador_perfis)
    │
    └─→ NOTIFY operador_updated, '{"user_id": "uuid"}'
             │
             └─→ NestJS PG pub/sub listener
                      │
                      └─→ DEL operador:profile:{user_id}
```

Este mecanismo garante que nunca existe "cache stale" após uma escrita — sem polling e sem TTL desnecessariamente curto. A separação de triggers por tabela permite invalidação granular: uma alteração em `operador_perfis` não invalida cache do cidadão e vice-versa.
