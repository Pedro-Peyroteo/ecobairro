#### Perfil do Cidadão (CRUD)

| #   | Método   | Rota                             | Descrição                                                 | Auth    | Fluxo                                               |
| --- | -------- | -------------------------------- | --------------------------------------------------------- | ------- | --------------------------------------------------- |
| P1  | `GET`    | `/cidadaos/me`                   | Obter perfil próprio                                      | CIDADAO | NestJS → Redis hit ou PG réplica (JOIN users + cidadao_perfis) |
| P2  | `PUT`    | `/cidadaos/me`                   | Actualizar campos não-sensíveis                           | CIDADAO | NestJS → PG write cidadao_perfis → NOTIFY cidadao_updated → Redis invalidar |
| P3  | `DELETE` | `/cidadaos/me`                   | Soft delete (direito ao esquecimento, RF-25, RNF-PRIV-03) | CIDADAO | NestJS → PG write users.eliminado_em → BullMQ anonymize job (limpa users + cidadao_perfis) |
| P4  | `GET`    | `/cidadaos/me/dados-sensiveis`   | Ver NIF/morada (só se consentimento activo)               | CIDADAO | NestJS → PG read cidadao_perfis (sem cache — PII)   |
| P5  | `PUT`    | `/cidadaos/me/dados-sensiveis`   | Actualizar NIF/morada (requer consentimento)              | CIDADAO | NestJS → valida consent → PG write cidadao_perfis cifrado |
| P6  | `GET`    | `/cidadaos/me/dashboard-widgets` | Obter configuração de widgets (RF-22)                     | CIDADAO | NestJS → Redis hit ou PG cidadao_perfis             |
| P7  | `PUT`    | `/cidadaos/me/dashboard-widgets` | Guardar ordem/selecção de widgets                         | CIDADAO | NestJS → PG write cidadao_perfis → Redis invalidar  |

**Resposta de P1 (perfil do cidadão):**

```
{
  id: uuid (users.id),
  email: string,
  phone: string | null,
  role: "CIDADAO",
  email_verified: boolean,
  nome_completo: string | null,
  zona_residencia_id: uuid | null,
  gamification_opt_in: boolean,
  notificacao_prefs: object | null,
  dashboard_widgets: object | null,
  criado_em: ISO8601
}
```

Nota: Os campos `nif_cifrado` e `morada_cifrada` NÃO são incluídos nesta resposta — são acedidos exclusivamente via P4 (`/dados-sensiveis`).

#### Admin — gestão de cidadãos

| #   | Método   | Rota                       | Descrição                | Auth  | Fluxo                                                   |
| --- | -------- | -------------------------- | ------------------------ | ----- | ------------------------------------------------------- |
| PA1 | `GET`    | `/admin/cidadaos`          | Listar cidadãos com filtros | ADMIN | NestJS → PG réplica (JOIN users + cidadao_perfis, nunca Redis — PII) |
| PA2 | `GET`    | `/admin/cidadaos/:id`      | Detalhe de um cidadão    | ADMIN | NestJS → PG réplica (JOIN users + cidadao_perfis)       |
| PA3 | `PUT`    | `/admin/users/:id/role`    | Alterar papel RBAC       | ADMIN | NestJS → PG write users.role → Redis invalidar user:rbac:{id} |
| PA4 | `DELETE` | `/admin/users/:id`         | Forçar anonimização      | ADMIN | NestJS → PG write users.eliminado_em + BullMQ           |

**Nota sobre PA3 (alteração de role):**
Quando o role muda entre tipos (ex: CIDADAO → OPERADOR_VEOLIA), o fluxo é:
1. Actualizar `users.role`
2. Criar o perfil na tabela de destino (`operador_perfis`) com dados mínimos
3. O perfil antigo (`cidadao_perfis`) é mantido para histórico mas fica inactivo
4. Invalidar `user:rbac:{id}` e `cidadao:profile:{id}` no Redis

#### Admin — gestão de operadores

| #   | Método   | Rota                         | Descrição                         | Auth  | Fluxo                                                   |
| --- | -------- | ---------------------------- | --------------------------------- | ----- | ------------------------------------------------------- |
| PA5 | `POST`   | `/admin/operadores`          | Criar conta de operador           | ADMIN | NestJS → PG write users + operador_perfis (transação)   |
| PA6 | `GET`    | `/admin/operadores`          | Listar operadores com filtros     | ADMIN | NestJS → PG réplica (JOIN users + operador_perfis)      |
| PA7 | `GET`    | `/admin/operadores/:id`      | Detalhe de um operador            | ADMIN | NestJS → PG réplica (JOIN users + operador_perfis)      |
| PA8 | `PUT`    | `/admin/operadores/:id`      | Atualizar perfil operador         | ADMIN | NestJS → PG write operador_perfis → Redis invalidar     |

**Corpo de PA5 (Criar operador):**

```
{
  email: string (required),
  password: string (required, min 10 chars),
  phone: string (optional),
  role: "OPERADOR_VEOLIA" | "TECNICO_AUTARQUIA" | "TECNICO_CCDR" | "ADMIN" (required),
  nome_completo: string (required),
  entidade_empregadora: string (required),
  cargo: string (optional),
  zonas_responsabilidade: uuid[] (optional, default [])
}
```

**Fluxo interno de PA5:**
1. Cria registo em `users` com o role especificado
2. Cria registo em `operador_perfis` com `user_id = users.id`
3. Ambos numa transação PostgreSQL
4. Envia email de verificação via BullMQ

**Filtros disponíveis em PA6:**

```
GET /admin/operadores?role=OPERADOR_VEOLIA&entidade=VEOLIA&zona_id=uuid&page=1&limit=20
```
